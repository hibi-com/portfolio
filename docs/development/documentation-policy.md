---

## 基本方針

- **変更が激しいものはドキュメントに書かない**: ディレクトリ構成・ソースの具体的な配置・ファイル一覧はすぐ陳腐化するため、リポジトリを直接参照する。
- **ドキュメントに書くもの**: 仕様（API・DB・エラーコード）、シーケンス（処理フロー）、ユーザーストーリー、設計思想（FSD/DDD の原則）、コマンド・手順・規約。
- **技術スタック**: 変わりやすいため [技術スタック](./tech-stack.md) の単一ファイルで管理し、他からは参照のみとする。

## 避けるべきパターン（アンチパターン）

| パターン | 理由 | 代わりに |
| -------- | ---- | -------- |
| ディレクトリツリーの列挙 | 追加・リネームで即ずれる | 「詳細はリポジトリの ○○/ を参照」とだけ記載 |
| ファイルパス・具体ファイル名の列挙 | 移動・リネームで即ずれる | 責務やレイヤー名など概念で説明 |
| 同一内容の二重管理 | CLAUDE.md と AGENTS.md などで内容が分岐する | 単一の情報源に寄せるか、片方から参照 |
| 命名規則の重複記載 | project-structure / feature-sliced / index など複数箇所でずれる | コーディング規約に一元化し、他からは参照のみ |
| 実装と異なる構造の図 | 図だけ更新忘れで誤解を招く | 図は概念レベルに留めるか、コードから生成する |
| パス・コマンドの細かい列挙 | ツールやスクリプト変更でずれる | 代表コマンドだけ記載し、詳細は `package.json` やリポジトリを参照 |

## 推奨

- 仕様書・シーケンス・ユーザーストーリーは `docs/` で管理し、実装前に参照する。
- ディレクトリやソースの「説明」が必要なときは、その場でリポジトリを開いて確認する習慣にする。
